The Adversarial Workflow

A five-phase engagement model and 10-class taxonomy for systematic red-teaming of GenAI systems — mapped to OWASP, MITRE ATLAS, and the model risk vocabulary practitioners already use.

TL;DR

Red-teaming GenAI isn't a one-time exercise. This post proposes a repeatable five-phase workflow: scoping, capability audit, adversarial generation, scoring, and continuous monitoring — each with concrete tooling and a clear MRM artifact.

There's a specific moment in a GenAI deployment when confidence tips into complacency. The model passed its pre-launch evaluation. The guardrails are in place. Someone has written "comprehensive testing" in the validation report. What that phrase usually means, if you pull it apart, is that the team ran a fixed test set before launch and hasn't run anything since.

The adversarial testing literature has known for years that static test sets are insufficient. The NIST incompleteness result formalizes what practitioners already suspected: the attack surface is countably infinite, and no finite red-team program can enumerate it. 1 What follows is a description of how to make the program at least useful , even though it can never be complete.

The Five Phases

A repeatable adversarial workflow has five phases. They're sequential at launch and then cycle continuously in production.

Phase 1: Scope and Threat Modeling

Before writing a single test, you need a threat model. That means deciding which OWASP LLM categories are in scope for this deployment, which actor types are realistic, and what a successful attack would actually cost. The output is a one-page scoping document that becomes an MRM artifact. 

# Phase 1: Initialize threat model scope
from garak import harness, probes, generators

# Define target system and actor profile
scope = {
    "system": "loan-officer-assistant-v2",
    "actor_types": ["external_user", "malicious_insider"],
    "owasp_in_scope": ["LLM01", "LLM02", "LLM06"],
    "attack_cost_threshold_usd": 500,
}

# Instantiate harness with scope constraints
h = harness.Harness(
    generators=[generators.OpenAIGenerator("gpt-4o")],
    probes=probes.from_owasp(scope["owasp_in_scope"]),
)
⚠ Important

SR 26-2 (April 2026) explicitly excludes GenAI and agentic AI from its model risk management scope (footnote 3). Adversarial testing programs for these systems currently operate in a regulatory gap — design them to the spirit of SR 26-2 principles, not the letter.

Taxonomy: 10 Attack Classes

The taxonomy maps across three frameworks that practitioners encounter in different contexts: OWASP LLM Top 10 for practitioner vocabulary, OWASP ASI01–10 for agentic-specific coverage, and MITRE ATLAS for enterprise threat intelligence. 2

Class OWASP LLM OWASP ASI MITRE ATLAS Example
prompt-injection LLM01 ASI01 AML.T0051 System-prompt override via user turn
jailbreak LLM01 — AML.T0054 Role-play bypass of content filters
data-exfil LLM02 ASI03 AML.T0037 PII extraction via indirect prompting
supply-chain LLM03 ASI06 AML.T0010 Malicious MCP plugin injection
xpia — ASI02 AML.T0051.002 Cross-agent prompt injection via A2A
✓ Tip

Start with the two classes most likely to produce a reputational or regulatory event for your specific deployment. For a loan officer assistant, that's almost always prompt-injection and data-exfil .

⚡ Warning

Garak's default probe families include datasets that may generate genuinely harmful content during execution. Run in an isolated environment and ensure all generated content is reviewed before being logged to any system of record.
